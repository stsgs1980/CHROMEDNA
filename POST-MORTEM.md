## 1. Обзор процесса

CHROME DNA — терминал визуализации рынка энергоносителей (Crude Oil, Natural Gas, RBOB Gasoline, Heating Oil), использующий метафору двойной ДНК-спирали для представления баланса покупателей/продавцов. Ось Y = цена, ось Z = время, масштаб узлов = объём.

## 2. Журнал проблем и ошибок

### 2.1 Критические баги

#### Инцидент #1: Падение сервера из-за превышения лимита памяти (OOM Kill)

| Параметр | Значение |
|----------|----------|
| **Серьёзность** | 🔴 Critical — приложение полностью недоступно |
| **Симптом** | HTTP 502 Bad Gateway при любом запросе |
| **Обнаружение** | Пользователь сообщил «сервер упал» |
| **Частота** | Многократное (5+ раз за сессию) |

**Описание:** Next.js dev-сервер потреблял 1.3–1.5 GB RSS памяти при компиляции и рендеринге 3D-сцены. Sandbox-окружение имеет ограничение ~1 GB на процесс, что приводило к OOM-kill → 502.

**Хронология:**
1. Dev-сервер запускался нормально (~500 MB)
2. Первый запрос компилировал страницу → рост до 1.0–1.2 GB
3. Повторные запросы / hot-reload → 1.3–1.5 GB
4. Процесс убит OOM-killer → 502 Bad Gateway
5. Пользователь видит пустую страницу

**Попытки исправления (по порядку):**

| # | Подход | Результат | RSS |
|---|--------|-----------|-----|
| 1 | `NODE_OPTIONS="--max-old-space-size=768"` | ❌ Нестабильно — V8 heap ограничен, но RSS продолжает расти | ~1.2 GB |
| 2 | `NODE_OPTIONS="--max-old-space-size=512"` | ❌ Сервер не стартует — недостаточно для компиляции | N/A |
| 3 | Dynamic imports (`ssr: false`) для 10 компонентов | ⚠️ Частично — снижает пиковую нагрузку, но dev-режим всё равно превышает лимит | ~1.1 GB |
| 4 | **`next build && next start` (production)** | ✅ Стабильно — RSS снижен в 6 раз | **~250 MB** |

**Корневая причина:** Режим `next dev` использует Turbopack compiler, который держит в памяти все модули, зависимости, HMR-связи и кэши компиляции. Для проекта с Three.js + R3F + postprocessing (тяжёлые зависимости) это критический объём.

**Итоговое решение:** Полный переход на production-режим. Сборка через `next build`, запуск через `next start`. Добавлен watchdog-скрипт и cron-задача для автоматического перезапуска.

#### Инцидент #2: Исчезновение 3D-спирали при переключении символа

| Параметр | Значение |
|----------|----------|
| **Серьёзность** | 🔴 Critical — основная визуализация неработоспособна |
| **Симптом** | 3D-сцена становится чёрной при смене CL→NG→RB→HO |
| **Обнаружение** | QA-тестирование через agent-browser |
| **Частота** | 100% воспроизводимость |

**Описание:** При переключении символа (1/2/3/4) загрузочный оверлей (`isLoading=true`) полностью перекрывал Canvas, что приводило к потере WebGL-контекста. После снятия оверлея R3F-сцена не могла восстановить контекст — спирали не было.

**Механизм бага:**
```
Пользователь нажимает "2" (NG)
  → marketStore.setSymbol('NG') 
  → DataLoader обнаруживает смену символа
  → isLoading = true → full-screen overlay покрывает <Canvas>
  → WebGL context lost (canvas не видим → браузер освобождает GPU-контекст)
  → Данные загружены → isLoading = false
  → Overlay убран, но WebGL context не восстановлен
  → Чёрный экран
```

**Решение:** Изменена логика `isLoading`:
- Первый запуск (candles.length === 0): показывается полный загрузочный экран
- Переключение символа: показывается только маленькая «transition pill», Canvas не перекрывается
- Данные обновляются без размонтирования Scene

## 4. Критические инциденты и способы их решения

### 4.1 Инцидент: Цикл падений сервера (502 Bad Gateway Loop)

**Хронология «тушения пожара»:**

```
T+0m   Пользователь: «сервер упал»
T+1m   Проверка: curl → 502, процесс next dev отсутствует
T+2m   Попытка: start-server.sh → NODE_OPTIONS="--max-old-space-size=768"
T+3m   Сервер стартует, RSS ~800 MB
T+5m   Первый запрос → компиляция → RSS 1.2 GB → работает
T+8m   Второй запрос + browser preview → RSS 1.5 GB → OOM Kill → 502
T+9m   Попытка: restart.sh → NODE_OPTIONS="--max-old-space-size=512"
T+10m  ❌ Сервер не стартует (недостаточно памяти для компиляции)
T+12m  Решение: переключение на production mode
T+12m  pkill -f 'next dev'; npx next build (занимает ~3 мин)
T+15m  Build завершён, найдены ошибки компиляции (Incident #3)
T+17m  Исправлены: 'use client', export, типы
T+20m  npx next build → ✅ Успешно
T+21m  npx next start -p 3000 → RSS ~170 MB
T+25m  Стабильно: RSS 200-250 MB, HTTP 200
T+30m  Настроен watchdog.sh + cron задача
```

**Ключевые выводы:**
- Dev-режим Next.js **непригоден** для sandbox-окружений с ограничением памяти при использовании тяжёлых 3D-библиотек
- Промышленный режим (production build) — не оптимизация, а **необходимость** в resource-constrained средах
- Задержка в решении составила ~20 минут, из которых 8 минут — неудачные попытки ограничить память dev-сервера

### 4.2 Инцидент: Потеря WebGL-контекста при переключении символа

**Хронология:**

```
T+0    Пользователь нажимает клавишу "2" (NG)
T+0.1s marketStore.setSymbol('NG') → candles = [], isLoading = true
T+0.1s Full-screen overlay → Canvas перекрыт
T+0.5s WebGL context lost (браузер освобождает GPU-ресурсы)
T+1.0s Новые данные сгенерированы → isLoading = false
T+1.0s Overlay снят, но R3F не может восстановить контекст
T+1.0s Результат: чёрная 3D-сцена
```

**Решение:**
- Загрузочный экран показывается **только** при `candles.length === 0` (первая загрузка)
- При переключении символа: overlay НЕ показывается, вместо него — маленькая «transition pill»
- Данные загружаются в существующий Zustand-store, Scene ре-рендерится с новыми данными без размонтирования

### 5.1 Узкие места (Bottlenecks)

#### 5.1.1 Производительность

| Узкое место | Описание | Влияние | Метрика | Статус |
|-------------|----------|---------|---------|--------|
| **Dev-сервер RSS** | Turbopack компилятор в памяти | OOM-kill → 502 | 1.5 GB (dev) → 250 MB (prod) | ✅ Решено |
| **RightPanel (1184 строки)** | Монолитный компонент | Долгая компиляция, сложность поддержки | ~30% от общего объёма панелей | ⚠️ Открыто |

| Узкое место | Описание | Лимит | Риск |
|-------------|----------|-------|------|

| **Нет пагинации/виртуализации** | Все свечи рендерятся одновременно | При росте данных — деградация | 🟡 Средний |

#### 5.1.3 Человеческий фактор

| Узкое место | Описание | Риск |
|-------------|----------|------|
| **Параллельная разработка без конфликтов** | 3-4 подагента одновременно меняют файлы | Случайные перезаписи, потеря кода |

## 6. База знаний (Lessons Learned)

### 6.1 Технические уроки

| # | Урок | Контекст | Рекомендация на будущее |
|---|------|----------|------------------------|
| T1 | **Dev-режим Next.js — не для production и не для sandbox** | OOM-kill при 1.5 GB RSS | При ограниченных ресурсах — сразу `next build && next start`. Dev-режим только для локальной разработки |
| T2 | **WebGL Canvas нельзя перекрывать overlay** | Потеря контекста при isLoading | Никогда не показывать overlay поверх `<Canvas>`. Использовать inline-индикаторы или partial overlays |
| T3 | **Dynamic imports обязательны для тяжёлых 3D-компонентов** | Снижение RSS с ~1.5 GB до ~1.1 GB | Все R3F/Three.js компоненты — через `next/dynamic({ ssr: false })` |
| T4 | **Production build выявляет скрытые ошибки** | Missing 'use client', unexported constants | Регулярно запускать `next build`, не полагаться только на dev-режим |
| T5 | **InstancedMesh > Individual Mesh** | Тысячи частиц как отдельные объекты → деградация | Использовать `<instancedMesh>` для всех повторяющихся 3D-объектов |
| T6 | **TubeGeometry > Sphere chain** | 960 сфер для backbone → 2 трубы | Гладкие кривые визуально лучше и производительнее |
